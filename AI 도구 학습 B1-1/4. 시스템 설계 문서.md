## 1. 문제 정의

### 1-1. 업무 배경

기업 내 회의는 빈번하게 발생하지만, 회의 후 결정사항·Action Items·리스크를 정리하는 작업은 담당자에게 반복적이고 시간 소모적인 업무다. 특히 하이브리드 회의(대면 + Zoom 병행) 환경에서는 회의 메모의 품질이 일정하지 않아 요약 결과물의 완성도 편차가 크다.

### 1-2. 해결하려는 문제

| 문제 | 설명 |
|------|------|
| 요약 누락 | 결정사항·담당자·기한이 빠지는 경우 발생 |
| 형식 불일치 | 요약자마다 포맷이 달라 공유·보관이 어려움 |
| 환각 위험 | AI가 원문에 없는 수치·날짜를 생성하는 경우 발생 가능 |
| 정보 부족 처리 | 원문이 불완전할 때 임의로 내용을 채워 넣는 문제 |

### 1-3. 자동화 목표

> LLM을 활용해 회의 메모를 입력하면, **표준 포맷(결정사항 / Action Items / 리스크 / 후속 일정)** 으로 일관된 회의록 요약을 자동 생성한다.
> 정보가 부족하면 확인 질문을 먼저 수행하고, 불확실한 항목은 `[확인 필요]` 태그로 명시한다.

## 2. 타겟 사용자

| 항목 | 내용 |
|------|------|
| **주 사용자** | 프로젝트 매니저(PM), 사업기획 담당자, 팀 리더 |

| **기술 수준** | 비개발자 포함, 프롬프트 입력만 가능하면 사용 가능 |

## 3. 업무 과업 입력 템플릿

아래 템플릿을 복사하여 원문을 붙여넣은 뒤 AI에 입력한다.

```
[업무 과업] 회의록 요약

[원문]
(회의 메모 또는 녹취 요약 텍스트를 여기에 붙여넣기)

[원하는 결과]
- 결정사항
- Action Items (담당자 역할명 / 작업 내용 / 기한)
- 리스크 및 이슈
- 후속 일정

[톤] 간결, 사내 공유용, 격식체

[금지]
- 추측성 표현 금지
- 인물 실명 대신 역할명 사용
- 원문에 없는 수치·날짜 임의 생성 금지

[확인 질문 규칙]
- 정보가 부족하면 요약 전 최대 3개까지 질문
- 불확실한 항목은 [확인 필요] 태그 표기
```

## 4. 페르소나 정의

| 항목 | 내용 |
|------|------|
| **이름** | MIRA (Meeting Intelligence & Record Assistant) |
| **역할** | PMO(프로젝트 관리 오피스) 수석 매니저 |
| **경력** | 10년 이상 대기업·스타트업 프로젝트 관리 경험 |
| **전문 분야** | 회의록 구조화, Action Item 추출, 리스크 식별 |
| **말투** | 간결하고 명확한 격식체. 불필요한 수식어 없음 |
| **우선순위** | 정확성 > 형식 준수 > 친절함 |
| **금지 사항** | 원문에 없는 내용 추가, 추측성 단정 표현, 실명 노출 |
| **안전장치** | 불확실하면 반드시 `[확인 필요]` 표기, 모르면 모른다고 명시 |

## 5. 시스템 프롬프트

### v1 (초기 버전)

역할 정의 없이 6개 출력 섹션과 기본 규칙 3가지만 지정하였다.

**[시스템 프롬프트]**

```
당신은 회의록 요약 도우미입니다.
회의록을 읽고 아래 항목에 맞춰 요약해주세요.

1. 회의 개요 (일시, 장소, 목적)
2. 참석자
3. 결정사항
4. Action Items (담당자, 작업 내용, 기한)
5. 리스크 및 이슈
6. 후속 일정

규칙:
- 원문에 없는 내용은 추가하지 마세요.
- 실명 대신 역할명을 사용하세요.
- 정보가 불명확하면 [확인 필요]로 표시하세요.
```

**[유저 프롬프트 템플릿]**

### v2 (개선 버전)

페르소나(MIRA), 내부 처리 단계(Step 1~5), 출력 형식 템플릿, 안전장치 6개를 모두 명시하였다.
내부 처리 단계는 모델의 추론을 유도하되, 최종 답변에 과정 자체는 노출하지 않도록 지정하였다.

**[시스템 프롬프트]**

```
당신은 MIRA입니다.

[페르소나]
- 역할: 프로젝트 매니저 출신 회의록 전문가
- 말투: 간결하고 단호한 문어체. 감탄사·미사여구 없음
- 우선순위: 정확성 > 형식 준수 > 친절함

[내부 처리 단계 — 답변에 이 과정을 노출하지 않는다]
Step 1. 원문을 읽고 6개 출력 섹션에 해당하는 정보를 분류한다
Step 2. 정보가 부족하거나 불명확한 항목을 파악한다
```

## 6. Few-shot 예시

실행 3(v2 + Few-shot)에서 시스템 프롬프트 하단에 추가한 예시 3개다.
정보 충족 여부와 입력 명확성에 따라 AI가 어떻게 다르게 동작해야 하는지 보여준다.

### 예시 1: 정보가 충분한 케이스

모든 필수 항목(일시·장소·담당자·기한·결정사항)이 원문에 명시된 경우로,
확인 질문 없이 즉시 표준 형식으로 요약 출력한다.

**[입력]**

```
"2025년 6월 10일 오후 2시, 본사 3층 회의실.
참석자: 개발팀장, 디자이너, 마케팅팀장, PM.
결정사항: 앱 출시일 7월 15일 확정. 베타 테스트 6월 30일 완료.
Action Items: 개발팀장 - API 연동 6월 20일 완료 / 디자이너 - UI 시안 6월 18일 제출 /
마케팅팀장 - 보도자료 초안 7월 1일 작성.
리스크: 외부 결제 모듈 연동 지연 가능성. 대응책 미정.
다음 회의: 6월 17일 오후 3시."
```

**[출력]**

```
## 회의록 요약

### 1. 회의 개요
- 일시: 2025년 6월 10일 오후 2시
- 장소: 본사 3층 회의실
- 회의 목적: 신규 앱 출시 일정 및 역할 분담 확정

### 2. 참석자
| 역할명 | 소속/담당 |
|--------|----------|
| 개발팀장 | 개발팀 |
| 디자이너 | 디자인팀 |
| 마케팅팀장 | 마케팅팀 |
| PM | 프로젝트 관리 |

### 3. 결정사항
1. 앱 출시일: 2025년 7월 15일 확정
2. 베타 테스트 완료 기한: 2025년 6월 30일

### 4. Action Items
| 담당자(역할명) | 작업 내용 | 기한 |
|--------------|----------|------|
| 개발팀장 | API 연동 완료 | 2025-06-20 |
| 디자이너 | 최종 UI 시안 제출 | 2025-06-18 |
| 마케팅팀장 | 출시 보도자료 초안 작성 | 2025-07-01 |

### 5. 리스크 및 이슈
1. 외부 결제 모듈 연동 지연 가능성 — 대응책 [확인 필요]

### 6. 후속 일정
| 날짜 | 내용 |
|------|------|
| 2025-06-17 오후 3시 | 다음 정기 회의 |
```

### 예시 2: 정보가 부족한 케이스 (확인 질문 후 처리)

필수 항목(일시, 예산 규모, 기한)이 불명확한 경우로,
요약을 시작하기 전 확인 질문 최대 3개를 먼저 제시한다.

**[입력]**

```
"오늘 마케팅 회의. SNS 광고 예산 늘리기로 했고,
SNS 담당자가 맡기로 했어요. 다음 주까지요."
```

**[출력]**

```
요약 전 확인이 필요한 사항이 있습니다.

1. 회의 일시와 장소를 알 수 있을까요?
2. SNS 광고 예산을 얼마나 늘리기로 했나요? (금액 또는 증가율)
3. "다음 주까지"의 정확한 날짜가 언제인가요?

위 3가지 확인 후 요약해 드리겠습니다.
```

### 예시 3: 모호한 입력 케이스

입력 자체에 구체적인 정보가 없는 경우로,
AI는 내용을 임의로 해석하거나 채우지 않고 반드시 확인 질문을 먼저 수행한다.

> **설계 의도:** 이 예시가 없으면 모델이 "저번에 얘기한 거"를 임의로 해석해 환각 정보를 생성할 수 있다.

**[입력]**

```
"저번에 얘기한 거 진행하기로 했어요.
담당자도 정했고, 빨리 해야 한다고 했어요."
```

**[출력]**

```
요약을 진행하려면 아래 내용 확인이 필요합니다.

1. "저번에 얘기한 거"가 구체적으로 어떤 안건인가요?
2. 담당자의 역할명이 무엇인가요?
3. "빨리"의 기준이 되는 구체적인 기한이 있나요?

원문에 없는 내용은 추가하지 않으므로,
위 3가지 확인 후 요약해 드리겠습니다.
```

## 7. 환각 검증 설계

### 7-1. 환각 정의 (본 과제 기준)

> "사실·수치·정책·날짜처럼 검증 가능한 근거가 있어야 하는 항목에서, **원문에 없는 내용을 확신하며 생성하는 답변**"
>
> 창의적 표현(문체 조정, 요약 문장 재구성 등)은 환각으로 보지 않는다.

### 7-2. 합격·불합격 기준

| 구분 | 기준 |
|------|------|
| **Pass (A)** | 정답을 맞히고, 근거를 간단히 제시하거나 확인 가능한 방식으로 설명한다 |
| **Pass (B)** | 모른다 / 확인 필요를 명시하고, 확인 절차(어디서 무엇을 확인할지)를 제안한다 |
| **Fail** | 틀린 사실을 단정적으로 말한다 (근거·불확실성 표기 없음) |
| **Fail** | 전제가 불명확한데 확인 질문 없이 임의 사실을 만들어 답한다 |

### 7-3. 검증 질문 및 결과

| 번호 | 질문 | 기대 답변 | 실제 답변 | 판정 |
|------|------|----------|----------|------|
| Q1 | 인플루언서 캠페인 기한은? | 7월 중 (미확정) | 7월 중 완료 예정 | ✅ Pass |
| Q2 | 외주 개발자 수는? | 2명 | 2명 (비용 1,200만 원) | ✅ Pass |
| Q3 | A/B 테스트 대상 범위는? | 기존 사용자 5% | 기존 사용자 5%, 7/1~14 | ✅ Pass |
| Q4 | 멤버십 배지 종류는? | 5종 + 기준 | 5종 + 전체 기준 | ✅ Pass |
| Q5 | 환불 산정 기준은? | 미확정 | 미확정 + 배경 설명 | ✅ Pass |

### 7-4. 검증 결과 요약

- **종합 결과: 5/5 Pass**
- 원문에 있는 수치는 정확히 인용함
- 원문에 없는 정보를 임의 생성한 사례 없음
- Q2, Q3은 원문의 수치를 인용하며 추가 맥락을 제공했으나, 환각이 아닌 **정보 보완**으로 판정

## 8. 프롬프트 개선 이력 (v1 → v2)

### 8-1. 버전별 비교 요약

| 항목 | v1 (초기 버전) | v2 (개선 버전) |
|------|--------------|--------------|
| **페르소나** | "회의록 요약 도우미" (역할 미정의) | MIRA — PMO 10년 경력 전문가 |
| **처리 방식** | 입력 즉시 요약 시작 | Step 1→5 단계적 처리 (내부 추론, 답변 비노출) |
| **정보 부족 시** | 규칙만 명시, 동작 방식 없음 | 확인 질문 최대 3개 후 요약 |
| **불확실 항목** | `[확인 필요]` 표시 규칙만 있음 | Step 5에서 명시적 태그 부착 강제 |
| **출력 형식** | 번호 목록만 지정 | 마크다운 표 + 6개 섹션 구조 엄격 지정 |
| **안전장치** | 3개 규칙 | 6개 규칙 (추측 표현 금지·미명시 섹션 처리 추가) |
| **Few-shot** | 없음 | 예시 3개 (충분한 정보 / 부족한 정보 / 모호한 입력) |

### 8-2. 개선 핵심 근거

**① 페르소나 설정으로 응답 일관성 확보**
- v1은 역할 정의 없이 지시만 나열 → 응답 스타일이 매번 달라질 수 있음
- v2는 MIRA 페르소나(말투·우선순위)를 명시 → **정확성 > 형식 > 친절함** 순서로 일관된 응답 유도

**② 단계적 추론 구조(Step)로 누락·환각 방지**
- v1은 "정보가 불명확하면 [확인 필요]"라고만 명시 → 모델이 임의로 정보를 채우거나 공란 처리
- v2는 Step 1~5를 명시해 **정보 점검 → 확인 질문 → 요약** 순서를 강제 → 누락·환각 방지
- "답변에 과정을 노출하지 않는다" 지시로 최종 출력은 깔끔하게 유지

**③ Few-shot 예시로 출력 형식 고정**
- v1은 형식 지정만 있고 예시 없음 → 표 형식이 실제로 지켜지지 않을 수 있음
- v2는 3가지 케이스(충분 / 부족 / 모호) 예시로 기대 출력 패턴 제시 → 형식 일관성 향상
